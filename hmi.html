Design Rationale — HMI Decisions
Design DecisionSRS / HazardRationale
Exactly 3 controls — drink type, volume, brew confirmSRS 2.1Minimises state space and error surface. No configuration menus. First-time unaided use validated at VE-B1.
Two-step brew confirmation — BREW then CONFIRM BREWH-19 child misuse, SRS 1.7Child pressing a single button cannot initiate a brew cycle with a hot fluid path. Deliberate second confirmation required.
Icon-based status ring — no text labelsSRS 1.7, SRS 2.1Multi-language hotel deployment. Text labels would require localisation for each market. Icon-only works across all languages.
7 distinct states — colour and text differentiatedSRS 1.7All 7 states are identifiable without documentation. Colour alone is not used as the sole differentiator (accessibility).
Volume slider 200–400 mlSRS 1.3, 4.6, 4.7Covers short americano to large travel mug format. Stepped at 50 ml for tactile precision without over-complicating the interface.
Real-time telemetry log — cloud ops sideSRS 1.9Guest never sees this. Cloud ops receives all events within 60 s. Proactive maintenance model depends on timely telemetry.
