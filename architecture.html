Governing Structuring Principles
Per Z8 — defined before any allocation decisions were made
Through Life Support
Every subsystem boundary, COTS choice, and interface decision is evaluated against its contribution to MTBF ≥26,280 h. Fan et al. failure data drives all high-risk component decisions.
Minimise interface count
SS-2 and SS-5 merged into a single Control Assembly at V2.0 — reduces internal interface count from 7 to 5, removing one class of failure mode (TD-04).
Hardware-layer safety
Safety-critical functions (TCO, PRV, WDT IC) are implemented in hardware independent of the MCU and firmware. No single software fault can defeat a safety function.
COTS where certified
IEC 60335, EMC, and food-safety certification is transferred to established COTS suppliers wherever possible. Bespoke design is reserved for CoffCo IP elements only.
Modular field replacement
All subsystems are assembly-swap replaceable. Field engineer restores service without disturbing adjacent assemblies. Target MTTR ≤4 h.
Implementation-neutral requirements
No consumable format (capsule, pod, bean) is named anywhere in SRS. Architecture and requirements are agnostic to the highest-leverage commercial decision.
Open Architecture Decisions
6 trade decisions — 4 closed, 2 open pending further evidence or stakeholder input
Ref Decision Status Detail
TD-01Consumable formatOpen — highest riskProprietary sealed format vs licensed third-party. Highest-leverage commercial decision. Drives supply chain, margin, and brand differentiation. Deferred to commercial team.
TD-02Thermal path topologyClosed — unifiedSingle unified thermal path with fluid-switching valve selected. Eliminates separate milk thermal path, reducing seal failure surface area by ~40% vs dual-path.
TD-03Regional power strategyClosed — plug moduleInterchangeable plug module (BS1363 / NEMA / CEE 7/16) on universal COTS PSU. Single PSU variant for all markets. No re-certification on market entry.
TD-04Control subsystem structureClosed — mergedSS-2 and SS-5 merged into single Control Assembly at V2.0. Reduces interface count from 7 to 5; eliminates one independent failure class.
TD-05Scale management strategyClosed — firmwareFirmware-scheduled descale cycle (SRS 1.6) based on brew count and hardness threshold. Alert transmitted 10 brews before maintenance required. No mechanical scale trap.
TD-06Wireless technologyOpen — IT survey neededWi-Fi vs BLE vs Zigbee. Requires hotel IT infrastructure survey from Vacation Stables. Drives SRS 1.9, 1.11 implementation. COTS pre-certified module will be used regardless of technology selected.

Fig 3 ▶
Behavioural — System State Machine
11 FSM states — 9 operational (inc. OTA-UPDATE) + 2 safety-priority · required by DoD 3
11-state finite state machine: 9 operational states plus the two Issue 2.1 safety-priority states. SAFE-SHUTDOWN and FLUID-EMPTY (red) supersede all other transitions: SAFE-SHUTDOWN de-energises all actuators on any SAF-01/07/11/12 trigger and recovers only via power cycle with field-engineer notification; FLUID-EMPTY inhibits the heating element on a brew command with level below minimum (SRS 1.10, SAF-02) and recovers automatically on refill. OTA-UPDATE entered from STANDBY only within the 03:00–06:00 window, with no brew activity for ≥30 consecutive minutes; automatic rollback to the previous A/B partition if STANDBY is not achieved within 120 s post-reboot (UPDATE-FAILED → FAULT). UPDATE-COMPLETE telemetry transmitted on success.

Behavioural — Brew Cycle Activity Sequence
Primary use case: americano and hot frothed milk · required by DoD 3
Brew cycle activity sequence from user selection through the mandatory pre-cycle check, heating, extraction, dispensing, and return to standby. The dry-fire prevention gate (SRS 1.10) is shown as a decision node before any heating element activation — if water or consumable levels are below minimum, heating is inhibited and a refill indication is displayed. Two-step confirmation (BREW then CONFIRM BREW, H-19) shown at initiation.
